# 4장. 독자 관점에서 릴리스 문서와 장애 보고서 쓰기

## 1. 체인지 로그를 분류, 요약, 종합하는 법

### 체인지 로그의 양과 만족도의 관계

일을 하고 나서 일의 내용을 상사나 고객에게 글로 보고해야 할 때 **체인지 로그**(changelog)를 사용한다. 이는 간단히 쓰면 한 일이 없어 보이고 자세히 쓰면 아무도 읽지 않게 된다.체인지 로그를 적절한 양으로 쓰려면 일단은 체인지 로그를 최대한 많이 써야 한다. 그다음에는 일정한 기준으로 선정하고 비슷한 것끼리 분류한 뒤 문장을 요약하는 기술이 필요하다.

- 1단계: 선정하기
- 2단계: 분류하기
- 3단계: 요약하기
- 4단계: 종합하기

### 1단계: 선정하기

체인지 로그 선정 기준을 개발자 기준으로 따지면 안 된다. 독자의 입장을 다를 수 있기 때문이다. 게다가 체인지 로그는 회사의 공식 발표이며, 독자와 직접 소통하는 문건이다. 다음 표로 체인지 로그 순위나 수준을 결정할 수 있다.
|회사가<br>말하고 싶다|개발자가<br>말하고 싶다|독자가<br>듣고 싶다|순위|수준|예시|
|-|-|-|-:|-:|-|
|O|O|O|1|자세히|인공지능 추천 추가|
|O|O|X|2|간단히|대용량 네트워크 증설|
|O|X|O|3|간단히|UI 개선|
|O|X|X|4|주석|면책 조항, 정책 변경|
|X|O|O|5|참고|작업 과정, 레퍼런스|
|X|O|X|삭제|-|성과가 낮은 오류 수정|
|X|X|O|삭제|-|경쟁력 없는 기능 추가|
|X|X|X|삭제|-|사소한 오류 수정|

### 2단계: 분류하기

공개할 체인지 로그를 선정했으면 비슷한 체인지 로그끼리 묶어야 한다. 이때 두 가지 방법이 있다.

- 첫 번째 방법은 개발 관점에서 비슷한 작업을 묶는 방법으로, 독자가 개발자인 경우에 유용하다.

```
- 새로운 기능 추가
    - 닉네임을 만들 때 특수 문자를 입력하는 기능을 추가했습니다.
    - 빈 게임방을 자동으로 검색하는 기능을 추가했습니다.
- 기능 개선
    - 용량이 큰 사진을 등록할 때 휴대전화 메모리 사용을 최소화하도록 등록 방식을 개선했습니다.
    - 최근 기록이 상위에 올라오도록 개선했습니다.
    - 게임 종료 후 바로 순위를 볼 수 있도록 개선했습니다.
- 오류 수정
    ...<이하 생략>
```

- 두 번째 방법은 독자가 일반 사용자인 경우에 유용하며, 사용자 관점에서 비슷한 것끼리 묶는 방법이다.

```
- 게임 준비
    - 미리 보기에서 간혹 리부팅되는 문제를 해결했습니다.
    - 빈 게임방을 자동으로 검색하는 기능을 추가했습니다.
- 게임 중
    - 고해상도 폰에서 아이콘이 찌그러지는 오류를 수정했습니다.
    - 가로/세로 화면 전환 시 하단 메뉴가 사라지는 오류를 수정했습니다.
- 게임 종료
    ...<이하 생략>
```

### 3단계: 요약하기

체인지 로그를 분류했으면 문장 단위로 요약할 차례다. 여러 가지 요약 방법 중에서 서술식 문장을 개조식 문장으로 바꾸는 방법을 쓰면 내용이 자연스럽게 축약되고 의미는 더 분명해진다. 개조식 문장으로 만들려면 불필요한 부사나 형용사, 조사나 어미를 없애고 정확하고 적절한 단어로 대체하면 된다.

```
- 게임 준비
    - 미리 보기 리부팅 문제 해결
    - 빈 게임방 자동 검색 기능 추가
- 게임 중
    - 고해상도 폰에서 아이콘이 찌그러지는 오류 수정
    - 가로/세로 화면 전환 시 하단 메뉴가 사라지는 오류 수정
- 게임 종료
    ...<이하 생략>
```

### 4단계: 종합하기

상사나 고객이 체인지 로그를 하나씩 뜯어보기 전에 이번 릴리스가 이전과 비교해서 뭐가 다르고 핵심과 컨셉이 무엇인지 한마디로 알려줘야 한다. 릴리스 내용 전체를 종합해서 한 문장으로 만들고 그것을 체인지 로그 첫 줄에 적어야 한다.

```
            사용자 편리성 개선

- 게임방에 더 빨리 입장
- 게임 결과 바로 확인

[세부 내용]
- 게임 준비
    - 미리 보기 리부팅 문제 해결
    - 빈 게임방 자동 검색 기능 추가
- 게임 중
    - 고해상도 폰에서 아이콘이 찌그러지는 오류 수정
    - 가로/세로 화면 전환 시 하단 메뉴가 사라지는 오류 수정
- 게임 종료
    ...<이하 생략>
```

## 2. 고객에게 유용한 정보를 쓰자

### 개발자 관점과 고객 관점

체인지 로그는 개발자가 변경한 내용을 적은 것. 하지만 독자는 뭔가 새로운 것, 바뀐 것, 명확한 행동의 지시를 보고 싶어 한다. 사내 개발자들끼리 본다면 한 일만 담백하게 적어도 상관없다. 외부 개발자나 일반 사용자가 보는 체인지 로그는 고객 관점에서 써야 한다.

`댓글에서 애니메이션 스티커 때문에 화면이 멈추는 문제를 해결했습니다.`

이것을 고객 관점에서 살펴보면 우선 고객의 불편이 뭔지 알아야 한다.

- 개발자의 문제: 화면이 멈춘 것
- 고객의 문제: 애니메이션 스티커를 댓글에 사용할 수 없는 것

개발자가 문제를 해결했다면 고객의 문제도 해결된다.

- 개발자의 문제 해결: 화면이 멈추지 않게 했다.
- 고객의 문제 해결: 애니메이션 스티커를 댓글에 사용할 수 있다.

이 내용을 합쳐서 체인지 로그 작성 시,

- `댓글에서 애니메이션 스티커 때문에 화면이 멈추는 문제를 해결했습니다. 이제 애니메이션 스티커를 정상적으로 댓글에 사용할 수 있습니다.`

고객의 문제 해결을 앞쪽으로 옮길 시,

- `이제 애니메이션 스티커를 정상적으로 댓글에 사용할 수 있습니다. 댓글에서 애니메이션 스티커 때문에 화면이 멈추는 문제를 해결했습니다.`

개발자의 문제를 짧게 줄일 시,

- `이제 애니메이션 스티커를 정상적으로 댓글에 사용할 수 있습니다.(화면 멈춤 문제 해결)`

개조식으로 바꿀 시,

- `애니메이션 스티커를 댓글에 정상 사용 가능(화면 멈춤 문제 해결)`

실제 사례로 네이버 밴드 애플리케이션의 변경 사항은 다음과 같다.

```
변경 사항

채팅 답장 기능이 추가되었어요!
상대방 메시지를 꾹 눌러 답장쓰기를 해보세요.

밴드의 일정 참석 요청을 놓치지 않으려면,
가입 밴드의 일정 푸시 알림을
[참석확인과 미리알림 받기]로 설정해보세요.
```

### 과거를 리뷰하고 미래를 보여주자

체인지 로그에는 '한 일'을 적는다. 하지만 개발자는 잡아야 할 버그가 아직 남아 있고 연구해야 할 개선 사항도 있고 개발할 새 기능도 많다. 고객 입장에서도 요구하거나 불평한 것이 반영되지 않으면 다음 릴리스를 기다릴 수밖에 없다.

다소 귀찮기는 하지만 다음 릴리스 항목으로 검토하는 것이 있다면 중요한 것은 공개하는 것이 좋다.
예를들면 카카오 맵의 체인지 로그다. '주요 개선 항목' 다음에 '검토 중인 항목'을 추가했다.

```
[주요 개선 항목]
- 지도 축소 상태에서 도로 굵기 개선, 지역 명청 컬러 개선
...<중략>

[검토 중인 항목]
- 건물 가독성 개선 검토
- 색각이상자 대응 검토
...<이하 생략>
```

### Semantic Versioning(유의적 버전)

체인지 로그는 버전 번호와 항상 같이 사용된다. 패치를 했거나 새로운 기능을 추가했거나 전체를 완전히 업그레이드했을 때 체인지 로그를 쓰면서 버전을 올린다.

- 1. 버전 1.2.2 -> 1.2.3
- 2. 버전 1.2.2 -> 1.3.0
- 3. 버전 1.2.2 -> 2.0.0

1번은 간단한 패치를 의미한다. 그래서 이전 버전과 호환된다.
2번은 새로운 기능을 추가했을 때다. 이전 버전과 일부 호환될 수 있으나 새로운 기능을 이전 버전에서 사용할 수 없다.
3번은 전면적인 업그레이드여서 이전 버전과는 호환되지 않는다고 볼 수 있다.

저자는 여기서 간단하고 쉽게 버전을 올릴 수 있는 가이드로 깃허브의 공동창업자 톰 프레스턴 베르너가 정한 Semantic Versioning을 제시했다. 자세한 내용은 [여기](https://semver.org/lang/ko/)를 참고하면 된다.

## 3. 릴리스 문서는 문제 해결 보고서처럼 쓰자

### 문제와 문제점을 구별하자

기업에 소프트웨어를 판매하거나 SI 사업을 한다면 릴리스 문서를 정식으로 만들어서 고객에게 제공해야 한다.
버그를 잡거나 새로운 기능을 추가하거나 성능을 개선하는 것은 어떤 문제를 해결하기 위해서다. 이때 문제는 목표와 현상의 차이다.

- 목표
  - 있어야 할 모습
  - 바람직한 상태
  - 기대되는 결과
- 현상
  - 현재의 모습
  - 예상되는 상태
  - 예기치 못한 결과

문제를 해결한다는 것은 목표에 다다르지 못하는 문제를 해결함으로써 현상을 목표에 일치시키는 것이다.
이러한 문제에는 세 가지 종류가 있다.

- 발생형 문제: 눈앞에 당장 발생해 당장 걱정하고 해결하기 위해 고민하는 문제, 프로그램 에러가 대부분 발생형 문제다.
- 탐색형 문제: 현재 상황을 개선하거나 효율을 높이는 문제다. 프로그램 개선이나 시스템 효율화가 대부분 탐색형 문제다.
- 설정형 문제: 미래 상황에 대응하는 문제다. 새로운 기능이나 대폭적인 업그레이드가 대부분 설정형 문제다.

버그 수정, 기능 및 성능 개선, 새로운 기능 제작은 모두 문제를 해결하는 것이다. 하지만 여기서 문제와 문제점을 구별해야 한다.

### 문제, 문제점, 해결책, 후속 계획 순으로 적자

하나의 문제에 문제점은 여러 가지이고, 여러 가지 문제점을 모두 해결하기에는 예산과 인력이 부족하여 특정 문제점을 선택할 수밖에 없다.
릴리스 문서는 결국 개발자가 문제점 하나를 선택해서 해결한 결과다. 따라서 여러 문제점 중에 어떤 문제점을 선택했는지를 독자에게 알려줘야 한다.

- 문제: 사용자가 급증하면 서버가 정지
- 문제점: 잘못된 시스템 설정, 프로그램 비 최적화, 잘못된 DB 설계
- 해결책: 시스템 설정 변경
- 후속 계획: 프로그램 최적화, DB 재설계

이것을 릴리스 노트에 요약하여 적으면 다음과 같다.

`OOO 서비스에 사용자가 급증하면 OO 서버가 정지하는 문제는 시스템 재설정으로 해결했습니다. 추후 프로그램 최적화와 DB 재설계도 검토하겠습니다.`

이처럼 문제, 문제점, 해결책, 후속 계획 순서로 쓰는 것이 좋다.
그래야 고객이 정확하게 내용을 파악할 수 있고 향후 변경 계획도 짐작할 수 있다.

### 법적인 문제를 고려해서 쓰자

릴리스 노트도 보고서의 일종인 만큼 보고서의 형식을 어느 정도 갖춰야 한다.

- 문서 정보: 제품명, 릴리스 이름, 릴리스 버전, 릴리스 날짜 등
- 개요: 릴리스 노트의 주요 내용을 종합한 글
- 새로운 기능: 이번 릴리스에 새롭게 추가한 기능
- 개선 사항: 기존 기능을 향상하거나 안정성 등을 강화한 내용
- 버그 수정: 버그 내용과 수정 사항
- 영향과 주의사항: 릴리스로 인한 영향과 개발자의 주의사항
- 연락처: 문의나 의견 접수를 위한 담당자 이름과 연락처 정보
- 면책: 변경 사항이나 릴리스 문서로 인한 법적 문제 발생 시 책임의 한계에 관한 내용

실제 재판에서 재판관이 릴리스 노트를 계약서에 준하는 문서로 판단할 수도 있다. 따라서 면책 조항을 꼭 적는 것이 좋다.
세 가지로 조항을 분류할 수 있다.

- 필수: 반드시 해야 할 일, 독자가 그 일을 하지 않아서 생기는 문제에 책임을 지지 않는다.
  - "~해야 한다."
  - "~하면 안 된다."
- 권장: 해결 방법이 많은 경우 우선순위를 개발자가 정해줄 때 쓴다. 권장하지 않은 방법을 써서 생기는 문제 역시 책임지지 않는다.
  - "~할 것을 권장한다."
  - "~하는 것이 좋다."
- 선택: 독자에게 여러 옵션을 자유롭게 줄 수 있을 때 쓴다.
  - "~할 수도 있다."
  - "~해도 된다."

그 외에도 문제가 생길 때를 대비한 예외 사항을 명시해야 면책이 될 수 있다.

## 4. 비즈니스를 이해하는 장애 보고서 쓰기

### 장애 보고서의 특징

- 첫째, 장애 보고서는 개발자가 원할 때 쓸 수 없다.
  - 미리 써놓거나 예상하고 쓸 수 없다.
  - 따라서 신속한 글쓰기가 필요하다.
- 둘째, 장애의 1차 원인은 대부분 다른 원인의 결과다.
  - 더는 원인의 원인을 찾을 수 없을 때 그 원인이 장애의 최초 원인이다.
  - 재발 방지를 위해 원인만 해결해선 안 된다. 발생 이유도 알아야 한다.
  - 원인과 이유를 찾는 분석적 글쓰기가 필요하다.
- 셋째, 장애 보고를 받는 윗사람은 대부분 개발자가 아니다.
  - "손해가 얼마인가요?"에 대처할만한 정도의 비즈니스 관점의 글쓰기가 필요하다.
- 넷째, 장애를 해결했다고 해서 100% 해결한 것은 아니다.
  - 윗사람은 확정적인 대답을 원하고 장애는 재발할 수 있다.
  - 정치적 글쓰기가 필요하다.
- 요약하면 다음과 같다.
  - 1. 질문에 대답하는 신속한 글쓰기
  - 2. 원인과 이유를 찾는 분석적 글쓰기
  - 3. 상사를 고려하는 비즈니스 관점의 글쓰기
  - 4. 원하는 것을 얻는 정치적 글쓰기

### 질문에 대답하는 신속한 글쓰기

대화를 글로 옮기는 방법이 있다. 일단 무슨 말이든 내뱉고 나서 생각을 정리하고,
질문을 받으면 답하면서 다음 생각으로 이어가는 방법이다.

- 동료와 질문을 나눈거처럼 말로 대답하듯이 작성한다.
- 대화 내용을 개조식으로 간단히 정리한다.
- 동료의 질문을 각 항목으로 분류할 수 있다.
- 항목과 내용을 장애 보고서 서식에 맞춰 작성한다.

### 원인과 이유를 찾는 분석적 글쓰기

문제 해결 기법으로 5Whys가 있다. 근본적인 원인을 찾는 기법으로, 문제의 원인이 되는 인과 관계를 탐색할 때 다섯 번 반복해서 원인이 무엇인지 질문하는 방식이다.

- Why의 뜻 두 가지
  - 1. 사물이나 현상, 동작이 문제를 초래하는 원인
  - 2. 사람이 문제를 초래한 이유

원인은 어떤 일이나 사건이고 이유는 사람이 어떤 일을 하거나 하지 않은 까닭이나 동기다. 따라서 Why는 어떤 일 또는 사람이다.

### 상사를 고려하는 비즈니스 관점의 글쓰기

장애 보고서를 비즈니스 관점으로 쓴다는 것은 이처럼 매출과 비용 관점으로 설명하는 것이다. 쇼핑몰 결제 기능이 작동하지 않은 예를 보면 개발자 관점과 비즈니스 관점은 다음과 같이 큰 차이가 있다.

- (개발자 관점) 장애로 인해 사용자가 1시간 동안 결제 못함 -> 기능 작동X
- (비즈니스 관점) 장애로 인해 OO억 원의 매출 손실이 발생함 -> 기대 매출 손실 발생

장애로 인한 손실을 계산하는 것이 곧 비즈니스 관점으로 장애를 보고하는 방법이다.

### 원하는 것을 얻는 정치적 글쓰기

확정적인 대답을 원하는 윗사람들에게 재발 가능성에 대해 퍼센티지로 얘기할 때 다음 표를 참고하여 확실하게 알려주는 것이 좋다.
| 장애 재발 가능성 | 표현 |
|-|:-|
|1%|절대 재발하지 않음|
|10%|재발하지 않음|
|20%|재발할지도 모름|
|30%|재발할 수도 있음|
|40%|재발한다고 볼 수도 있음|
|50%|재발할 수 있음|
|60%|재발하지 않는다고 볼 수 없음|
|70%|재발한다고 보여짐|
|80%|재발함|
|90%|재발할 것이 틀림없음|
|99%|반드시 재발함|

데이터 센터 냉방기 노후화의 보고서에도 다음과 같이 쓰면 도움될 수 있다.

- 변경 전
  - "냉방기 미교체 시 화재 발생 염려"
- 변경 후
  - "냉방기 폭발 시 서비스 전면 중단"
  - "냉방기 폭발 시 대표이사 구속 100%"
